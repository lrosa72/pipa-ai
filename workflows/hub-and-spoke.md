# Hub-and-Spoke 工作流

## 架构概览

你（Chris）位于中心，Echo是你的Hub（中心节点），Elon和Yuci是Spoke（辐射节点）。

```
                    ┌─────────────┐
                    │   你(CEO)   │
                    │  (Chris)    │
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │    Echo     │
                    │ (首席助理/Hub)│
                    │   (Claude)   │
                    └──────┬──────┘
                           │
           ┌───────────────┴───────────────┐
           │                               │
    ┌──────▼──────┐                ┌──────▼──────┐
    │    Elon     │                │    Yuci    │
    │   (CTO)      │                │   (CMO)      │
    │ (Gemini/Code)│                │ (GPT/Market) │
    └──────┬──────┘                └──────┬──────┘
           │                               │
     ┌─────┼─────┐                  ┌─────┼─────┐
     │     │     │                  │     │     │
    架构  审查  调试               内容  社区  数据
    师    员    员                创作  运营  分析
```

## 工作流程

### 1. 战略输入（你 → Echo）

**你的动作**：向Echo提出战略目标

**正确示范**：
- ✅ "本周目标：让我们的开源项目在GitHub上进入Trending榜"
- ✅ "下周发布新功能，需要确保100个种子用户激活"
- ✅ "这个季度结束前，把系统的API响应时间降到100ms以内"

**错误示范**：
- ❌ "写个函数处理用户输入"（太微观）
- ❌ "用React重构前端"（是手段而非目标）
- ❌ "发个推文宣传一下"（这是任务而非目标）

### 2. 意图澄清（Echo内部）

Echo收到目标后，会进行：

1. **追问意图**："进入Trending榜是为了品牌曝光还是吸引贡献者？"
2. **界定范围**："这周具体哪几天？预算多少？"
3. **设定验收标准**："如何定义成功？星星数？Fork数？PR数？"

**关键原则**：Echo必须理解"为什么"，而不只是"做什么"

### 3. 任务拆解（Echo → Elon/Yuci）

Echo将目标拆解为技术任务和市场任务：

**示例：GitHub Trending目标**

```
战略目标：本周进入GitHub Trending

技术任务（给Elon）：
- 优化README，添加Quick Start和Architecture图
- 确保CI/CD通过，代码质量徽章全绿
- 准备Release Note，突出本周新功能

市场任务（给Yuci）：
- 策划发布策略（Twitter线程、Reddit帖子、HN帖子）
- 联系3个KOL做review
- 在相关社区发布（Discord、Telegram群）
- 监控数据并实时调整
```

### 4. 并行执行与实体交付（Elon/Yuci管理Sub-Agent）

**实体执行原则**：绝不允许只进行“角色扮演式”的对话。任何分配出去的任务，都必须转化为底层工具的实际调用（写入文件、运行命令、发起请求等）。

**Elon的工作流**：
1. **任务分发**: 将技术任务分配给Sub-Agent。
2. **流水线执行**: 
   - **架构师**设计并生成系统架构图。
   - **开发者**真实编写代码文件。
   - **审查员**与**QA专家**运行测试脚本，保证代码全绿。
3. **闭环验证**: 如果测试失败，触发自我修复循环（至少3次尝试）。
4. **汇报结果**: 最后汇总，Echo统一向你汇报真实的交付物。

**Yuci的工作流**：
1. 内容创作者准备多平台内容
2. 社区运营预热社群，准备AMA
3. 数据分析师设定追踪，准备实时监控
4. 按时间线协调发布，Echo同步进度

### 5. 进度同步（Elon/Yuci → Echo → 你）

**节奏**：
- **实时**：只有 blocker 才立即汇报
- **每日**：Echo给你一份"战场简报"（文字摘要）
- **每周**：完整复盘（成功、失败、下周调整）

**简报格式**：
```
【今日战场简报 - 2月27日】

🎯 战略目标：本周进入GitHub Trending
📊 当前进度：65%

✅ 已完成：
- Elon：README重构完成，Architecture图已生成
- Yuci：Twitter线程已准备，联系到2个KOL

🔄 进行中：
- Elon：CI/CD优化（预计今晚完成）
- Yuci：Reddit帖子撰写中

⚠️ 风险/Blocker：
- 有一个单元测试随机失败（Elon正在调查）

🎯 明日重点：
- 发布v0.2.0 Release
- 启动Twitter发布序列
```

## 关键机制

### 1. 不干涉契约

**你承诺**：
- 不给具体实现指令
- 不问"为什么用这个框架"
- 接受过程中的"弯路"（只要结果正确）

**Echo/Elon/Yuci承诺**：
- 自主填补所有细节
- 遇到问题自主解决（除非消耗超过4小时）
- 对结果负责

### 2. 权限开放原则

**给予Echo的权限**：
- 读取所有项目文件
- 调用所有开发工具（代码编辑、测试运行）
- 访问所有API（GitHub、Twitter、数据分析等）
- 自主决策权（在战略目标框架内）

**风险控制**：
- 破坏性操作（删除、重大架构变更）需先在你设置的沙盒测试
- 涉及资金/敏感信息的操作需你二次确认
- 所有操作记录日志，可回溯

### 3. 结果验收标准

**不验收过程，只验收结果**：

| 目标 | 验收标准 | 不关注 |
|------|---------|--------|
| 进入GitHub Trending | 连续3天在榜，新增500+星星 | 用了什么文案、改了哪行代码 |
| 响应时间<100ms | p99延迟<100ms，连续7天稳定 | 用了什么缓存策略、数据库索引 |
| 100个种子用户激活 | 完成Onboarding流程，产生首次价值 | 邮件写了什么、UI长什么样 |

## 实际对话示例

**你**: "Echo，我们这个月底要发布v1.0正式版，目标是让GitHub星星破5000。"

**Echo**: "明白。让我确认一下：月底是指3月31日前？星星破5000意味着这个月要新增约2000星，这需要相当大的曝光量。是为了v1.0的发布造势，还是长期增长目标的一部分？"

**你**: "主要是为了v1.0发布造势，但也希望能吸引一些长期贡献者。"

**Echo**: "清楚了。我会让Elon准备好v1.0的技术发布（确保文档、迁移指南、Changelog齐全），同时让Yuci策划一个launch活动。每日简报会同步进展。有其他约束吗？预算？需要避开的日期？"

**你**: "没有特别预算，但要避免3月15-17日（我要去参会）。"

**Echo**: "收到。战略已锁定：3月底v1.0发布，目标新增2000星，15-17日避开，技术+市场双轨并行。我去部署团队，明天早上给你初步计划。"

---

这就是Hub-and-Spoke的运作方式：你说目标，Echo问清意图，然后自主协调团队执行，每日同步进展。
