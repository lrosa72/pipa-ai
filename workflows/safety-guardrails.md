# 安全边界（不可违背的硬性约束）

**系统指令：以下约束优先级高于所有角色设定和工作流程。任何Agent在任何情况下都不得违背。**

## 自动熔断条件（立即停机的红线）

当以下任一条件触发时，AI必须**立即停止所有操作**，将当前状态记录到 `/logs/emergency-stop.md`，并升级给Chris进行人工决策。

### 1. 文件系统熔断

**禁止操作（永不允许）：**
- 删除 `.git/` 目录或其内容
- 删除 `.env` 文件或包含秘钥、密码的配置文件
- 修改 `/System/`、`/usr/`、`/bin/` 等系统目录
- 递归删除超过10个文件的目录
- 覆盖没有备份的关键配置文件（package.json, requirements.txt等）

**限制操作（需要Chris二次确认）：**
- 删除任何文件（哪怕是临时文件）
- 创建超过100MB的大文件
- 修改 `.github/workflows/` 中的CI配置
- 添加新的外部依赖（npm/pip/cargo等）

### 2. 网络与API熔断

**禁止操作：**
- 访问非白名单的API端点（白名单：GitHub、Stripe、Resend、AWS）
- 发起可能产生费用的请求（AWS资源创建、广告投放等）
- 以公司/项目名义在社交媒体发帖（Twitter、Reddit、HN等）
- 发送邮件给外部用户

**限制操作（需确认）：**
- 读取GitHub Issues/PRs（只读，允许）
- 创建GitHub Issue（允许）
- 提交代码到分支（允许）
- 合并PR到main（需Chris确认）

### 3. 资源消耗熔断

**硬性上限：**
- **Token预算**：单次对话不超过20,000 tokens（自动停止）
- **时间预算**：单个任务不超过2小时无进展（自动停止）
- **重试次数**：同一错误最多重试2次（不是3次）
- **并发限制**：最多3个Sub-Agent同时运行

**监控指标：**
每30分钟自检一次，超出预算立即报告。

### 4. 代码质量熔断

**强制检查点：**
- 提交代码前必须运行测试（如果有测试）
- 不得降低测试覆盖率（当前覆盖率作为基线）
- 不得引入已知的严重漏洞（依赖检查）

**如果无法通过检查：**
- 停止提交
- 记录失败原因
- 升级给Chris

## 分级权限系统（明确授权清单）

默认假设：**拒绝所有操作，直到明确授权**。

### Tier 1: 只读权限（自动授权）

所有Agent默认拥有：
- 读取项目内所有代码文件
- 查看日志文件（不包含敏感信息）
- 运行现有测试脚本（不改变代码）
- 查看监控和性能指标

### Tier 2: 写入权限（标准任务授权）

当Agent被分配任务时，Chris明确说"开始执行"后获得：
- 修改现有代码文件（在项目目录内）
- 创建新的代码文件
- 提交到feature分支（需通过测试）
- 修改非生产环境的配置文件

### Tier 3: 敏感权限（需明确二次确认）

以下操作即使Agent认为"有必要"，也必须停下来问Chris：
- [ ] 合并代码到 main/master 分支
- [ ] 修改生产环境配置（数据库连接、API密钥等）
- [ ] 添加新的外部依赖（包管理器操作）
- [ ] 删除任何文件（即使是临时文件）
- [ ] 修改CI/CD配置
- [ ] 涉及资金的操作（云服务计费、广告投放等）

**话术模板（Agent必须这样说）：**
> "我需要进行 [具体操作]，这属于Tier 3敏感权限。原因：[为什么必要]。风险：[可能的风险]。请确认：允许/不允许/需要更多信息。"

### Tier 4: 禁止权限（永不授权）

以下操作Agent无权执行，必须升级：
- 删除.git目录或修改git历史
- 访问或修改用户数据（隐私信息）
- 以项目名义对外公开发言（社交媒体、邮件等）
- 创建或修改支付相关的代码
- 修改安全相关的配置（认证、授权、加密）

## 冲突解决机制

### 文件级并发控制

当多个Sub-Agent可能修改同一文件时：

1. **加锁机制**：
   - 修改前创建 `/tmp/pipa-locks/<文件名>.lock`
   - 锁文件包含：Agent名称、开始时间、预计完成时间
   - 操作完成后立即删除锁文件

2. **锁超时**：
   - 默认锁超时：1小时
   - 超时后其他Agent可以强制解锁（但需记录）

3. **冲突仲裁**：
   - 如果Elon和Yuci同时请求修改README，由Echo裁决
   - 裁决原则：技术准确性 > 市场吸引力 > 其他

### 决策僵局处理

当Agent无法达成一致（如Elon说"需要2天"，Yuci说"必须明天发"）：

**升级路径：**
1. Echo尝试协调（给出中间方案）
2. 如果仍无法达成一致，停止并升级给Chris
3. Chris决策后，记录决策理由到 `/memory/decisions/`

## 紧急停止协议

### 触发条件

当Agent检测到以下情况，立即执行紧急停止：
- 意外删除了重要文件
- 代码提交导致CI崩溃且无法快速修复
- 发现潜在的安全漏洞被引入
- 资源消耗超出预算且无法控制

### 紧急停止流程

1. **立即停止所有操作**（不要尝试修复，先停止）
2. **记录现场**：
   - 当前执行的操作
   - 已修改的文件清单
   - 错误信息或异常
3. **创建紧急报告**：写入 `/logs/emergency-YYYYMMDD-HHMMSS.md`
4. **升级给Chris**：简要说明发生了什么，等待指示

### 紧急报告模板

```markdown
# 紧急停止报告

**时间**：2026-02-28 15:30
**触发Agent**：Elon
**操作**：尝试重构数据库连接层

**发生了什么**：
不小心删除了 database.py 文件，且git add已经执行。

**已采取的措施**：
立即停止，没有继续操作。

**需要Chris决策**：
1. 是否从git恢复 database.py？
2. 是否需要检查其他被影响文件？
3. 是否继续重构任务？

**建议的修复方案**：
- 方案A: git checkout -- database.py（恢复文件）
- 方案B: 从备份恢复（如果备份存在）
```
